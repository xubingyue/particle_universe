Particle Renderer

Definition

The syntax of a Particle Renderer definition is:

renderer   rendererType   {renderer properties}

renderer is the keyword.
rendererType is the type of renderer.
The renderer properties are handled in the paragraph 'Properties' of each specific Renderer.

Example: 

renderer                 Billboard
{
   ...
}

Specific Renderers

  • Beam Renderer
  • Billboard Renderer
  • Box Renderer
  • Entity Renderer
  • Light Renderer
  • Ribbon Trail Renderer
  • Sphere Renderer

Base Properties


Render Queue Group

Instead of using a default render queue group, an specific queuegroup can be used. Learn more about render queue groups in the Ogre documentation. Syntax:

render_queue_group   #value

render_queue_group is the keyword.
#value is a uint8 value that identifies the render queue group. Default: 50 (RENDER_QUEUE_MAIN)

Sorting

Sort particles. Sorted billboards for instance can look better in some cases. Note, that sorting decreases the performance / framerate. For some renderers it doesn't really make sense to sort. This is usually the case for renderers that render 3D particles, such as the Box Renderer. Syntax:

sorting   true | false

sorting is the keyword.
true | false. Determines whether the particles must be sorted (true) or not. Default: false

Texture Coordinate Rows

Defining texture coordinates in a Particle Renderer serves several purposes. It can be used in combination with animated textures, where one texture contains several images (atlas texture). It is also possible to emit particles with an unique image, while only one texture is used. The bottomline is, that using one material (texture) with multiple images is 'cheaper' than using multiple materials. If you want to use texture coordinates, there are various ways how to do this. If the atlas texture contains images that are equally sized, the easiest way to define texture coordinates is to devide the texture in a number of rows and columns. This property defines the number of rows.
Each texture coordinate set gets an index, starting with 0, which identifies the top-left corner and ending with a rown * columns - 1, which identifies the bottom-right corner. Syntax:

texture_coords_rows   #value

texture_coords_rows is the keyword.
#value is a value > 0 that identifies the number of rows in a texture. Default: 1

Texture Coordinate Columns

This property is used in combination with the 'texture_coords_rows' property and defines the number of columns in a texture. Syntax:

texture_coords_columns   #value

texture_coords_columns is the keyword.
#value is a value > 0 that identifies the number of columns in a texture. Default: 1

Texture Coordinate Set

If an atlas texture contains images of various size and you want to use texture coordinates, you have to set them manually. Each added 'texture_coords' entry gets an index that starts with 0 and increments with each added entry. Syntax:

texture_coords_set   [texture_coords_define   #u   #v   #width   #height]

texture_coords_set is a keyword and defines the set.
texture_coords_define is a keyword and defines individual texture coordinates. One texture_coords_set contains one or more texture_coords_define.

#u is a value between [0..1] and defines the u-value in the texture.
#v is a value between [0..1] and defines the v-value in the texture.
#width is a value between [0..1] and defines the width of an image in the texture.
#heigh is a value between [0..1] and defines the height of an image in the texture.

Example: 

renderer Billboard
{
    texture_coords_set
    {
        texture_coords_define 0 0 0.0625 1
        texture_coords_define 0.0625 0 0.0625 1
        texture_coords_define 0.125 0 0.0625 1
        texture_coords_define 0.1875 0 0.0625 1
        texture_coords_define 0.25 0 0.0625 1
        texture_coords_define 0.3125 0 0.0625 1
        texture_coords_define 0.375 0 0.0625 1
        texture_coords_define 0.4375 0 0.0625 1
        texture_coords_define 0.5 0 0.0625 1
        texture_coords_define 0.5625 0 0.0625 1
        texture_coords_define 0.625 0 0.0625 1
        texture_coords_define 0.6875 0 0.0625 1
        texture_coords_define 0.75 0 0.0625 1
        texture_coords_define 0.8125 0 0.0625 1
        texture_coords_define 0.875 0 0.0625 1
        texture_coords_define 0.9375 0 0.0625 1
    }
}
Use soft particles

Soft particles are particles that fade if they approach another surface. This prevents sharp edges between the particles and other objects. Important to know when you use them:

  • This effect takes al lot of resources, because the scene is also rendered to a render texture.
  • Set the main camera (*) in the particle system if you can.
  • Soft particles donÂ´t work with the EntityRenderer. Billboard and Box renderers are tested and do work.

    (*) This is the camera that is used for creating the depthmap, needed for the effect.
    If the main camera is omitted, the first camera that is encountered is used.

Syntax:

use_soft_particles   true | false

use_soft_particles is the keyword.
true | false indicates whether soft particles are use (true) or not. Default: false

Contrast Power (soft particles)

Contrast Power defines the smoothness of the fading effect. Syntax:

soft_particles_contrast_power   #value

soft_particles_contrast_power is the keyword.
#value defines the smoothness of the fading effect. Default: 0.8

Scale (soft particles)

The scale defines the slope of the alpha fading and determines how fast alpha fading happens. Syntax:

soft_particles_scale   #value

soft_particles_scale is the keyword.
#value is the scaling amount. Default: 1.0

Delta (soft particles)

The delta value determines the difference between the depth of the scene and the depth of the particles. Beyond this difference, no alpha fading is applied anymore. Delta is a value in the range [-1, 0]. Syntax:

soft_particles_delta   #value

soft_particles_delta is the keyword.
#value is a threshold value that defines the difference between the depth of the scene and the depth of the particles. Default: -1.0
